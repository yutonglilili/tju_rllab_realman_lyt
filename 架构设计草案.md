# 多线程 Pick-and-Place 架构设计方案

## 一、总体思路（一句话）

**指令拆解 → 逐任务执行（多线程感知+规划+执行）→ 成功/失败判断 → 循环推进**

系统包含三个并行线程（感知、规划、执行），通过共享状态和事件机制协调工作。感知线程持续以固定频率对目标打点；当检测到物体位置变化或首次获取点位时，触发规划线程调用 curobo 生成动作序列；执行线程从动作列表中依次取点执行。动作列表执行完毕后，系统切换到"结果检测模式"，调用 VLM 判断 pick/place 是否成功，根据结果决定下一步动作。

---

## 二、系统总流程

```
程序启动
  │
  ├── 1. 初始化环境（RealmanEnv, 相机, curobo）
  │
  ├── 2. 调用 VLM 拆解指令 → 得到 task_list: [{pick, place}, ...]
  │
  └── 3. 遍历 task_list，对每个 task:
          │
          ├── 3.1  PICK 阶段
          │   ├── 子阶段1 APPROACH: 三线程协作（感知✅规划✅执行✅）
          │   │   └── 到达物体上方 pre_pick 位置（物体移动时自动重规划）
          │   ├── 子阶段2 CRITICAL: 仅执行（感知⛔规划⛔）
          │   │   └── 主线程阻塞执行：下压 → 闭合夹爪 → 提起
          │   └── 结果检测 CHECKING: 调用 VLM 判断 pick 是否成功
          │       ├── pick 成功 → 进入 PLACE 阶段（3.2）
          │       └── pick 失败 → 重试 PICK（3.1）
          │
          ├── 3.2  PLACE 阶段
          │   ├── 子阶段1 APPROACH: 三线程协作
          │   │   └── 到达容器上方 pre_place 位置
          │   ├── 子阶段2 CRITICAL: 仅执行
          │   │   └── 主线程阻塞执行：下放 → 松开夹爪 → 抬起
          │   └── 结果检测 CHECKING: 调用 VLM 判断 place 是否成功
          │       ├── place 成功 → 机械臂 reset，同时开始下一任务感知
          │       └── place 失败 → 重新回到 PICK 阶段（3.1）
          │
          └── 3.3  所有任务完成
```

## 三、核心数据结构：SharedState（线程间共享状态）

```python
import threading
import numpy as np
from enum import Enum, auto

class TaskPhase(Enum):
    """当前任务所处阶段"""
    PICK = auto()       # 正在执行抓取
    PLACE = auto()      # 正在执行放置

class SystemMode(Enum):
    """系统工作模式"""
    TRACKING = auto()       # 持续打点追踪模式（感知线程活跃）
    EXECUTING = auto()      # 动作执行模式（执行线程活跃）
    CHECKING = auto()       # 结果检测模式（检测成功/失败）
    RESETTING = auto()      # 机械臂复位模式
    IDLE = auto()           # 空闲

class SharedState:
    def __init__(self):
        self.lock = threading.Lock()

        # ===== 任务信息 =====
        self.current_task = None               # 当前任务 {'pick': ..., 'place': ...}
        self.task_phase = TaskPhase.PICK        # 当前阶段
        self.system_mode = SystemMode.IDLE      # 系统模式

        # ===== 感知输出 =====
        self.target_name = None                 # 当前追踪的目标名称
        self.latest_point_2d = None             # 最新的 2D 打点结果
        self.latest_point_3d = None             # 最新的 3D 坐标 (Base 坐标系, xyz)
        self.latest_target_T = None             # 最新的目标 4x4 位姿矩阵
        self.last_stable_point_3d = None        # 上一次稳定的 3D 坐标（用于变化检测）
        self.point_changed = False              # 标记：点位是否发生明显变化
        self.is_first_point = True              # 标记：是否为第一个有效点

        # ===== 规划输出 =====
        self.plan_stage = "APPROACH"            # 当前规划阶段: "APPROACH"（规划线程只在此阶段工作）
        self.action_list = []                   # 动作序列 [{"joints":..., "gripper":...}, ...]
        self.action_index = 0                   # 当前执行到的索引
        self.plan_ready = threading.Event()     # 规划完成信号
        self.need_replan = threading.Event()    # 需要重新规划信号

        # ===== 执行控制 =====
        self.execution_done = threading.Event() # 动作列表执行完毕信号
        self.abort_execution = threading.Event()# 中止当前执行信号

        # ===== 全局控制 =====
        self.stop_all = threading.Event()       # 全局停止信号
        self.pause_perception = threading.Event()  # 暂停感知线程信号（用于检测阶段）
```

## 四、三个核心线程

### 4.1 感知线程（Perception Thread）

**职责**：以固定频率调用 VLM 对当前目标物体打点，输出 2D/3D 坐标。

**两种工作子模式**（通过 `system_mode` 控制）：

| 子模式 | 触发条件 | 行为 |
|--------|----------|------|
| **TRACKING（持续追踪）** | 系统进入 TRACKING 模式 | 持续以固定频率打点，检测物体位置变化 |
| **暂停** | `pause_perception` 被设置 | 暂停打点（如进入 CHECKING 模式时） |

```
感知线程主循环:
┌─────────────────────────────────────────────────────────────────┐
│  while not stop_all:                                            │
│                                                                 │
│    if pause_perception.is_set():                                │
│        sleep(0.05)       # 暂停状态，等待恢复                     │
│        continue                                                 │
│                                                                 │
│    1. 获取 RGB 图像                                              │
│    2. 调用 VLM 打点: get_point_vllm(image, target_name)         │
│    3. 2D→3D 转换: make_target_T(...)                            │
│    4. 变化检测:                                                  │
│       ├── 如果 is_first_point (第一个点):                         │
│       │   ├── 保存为 last_stable_point_3d                        │
│       │   ├── 设置 is_first_point = False                        │
│       │   ├── 设置 point_changed = True                          │
│       │   └── 触发 need_replan 信号                              │
│       │                                                         │
│       └── 如果不是第一个点:                                       │
│           ├── 计算 dist = |new_xyz - last_stable_point_3d|       │
│           ├── 如果 dist > MOVE_THRESHOLD (如 3cm):               │
│           │   ├── 更新 last_stable_point_3d                      │
│           │   ├── 设置 point_changed = True                      │
│           │   ├── 触发 abort_execution (中止当前执行)             │
│           │   └── 触发 need_replan 信号                          │
│           └── 否则: point_changed = False (物体静止)              │
│                                                                 │
│    5. sleep(PERCEPTION_INTERVAL)  # 如 0.2s ~ 0.5s              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键设计：感知线程何时暂停/恢复**

```
                    ┌──────────────────────┐
                    │   系统进入 TRACKING   │  ← 感知线程活跃
                    │   持续打点追踪        │
                    └──────────┬───────────┘
                               │ 动作列表执行完毕
                               ▼
                    ┌──────────────────────┐
                    │   系统进入 CHECKING   │  ← 设置 pause_perception
                    │   暂停感知            │     感知线程暂停
                    │   调用 VLM 检测结果   │
                    └──────────┬───────────┘
                               │ 判断结果
                    ┌──────────┴───────────┐
              成功且需要继续          失败 → 重试
                    │                      │
                    ▼                      ▼
            清除 pause_perception    清除 pause_perception
            设置新目标               保持/恢复目标
            进入 TRACKING            进入 TRACKING
            感知线程恢复             感知线程恢复
```

### 4.2 规划线程（Planning Thread）

**职责**：当感知线程检测到目标位置变化（或首次获取到点位）时，调用 curobo 进行路径规划，生成 xyzrpy 动作序列。

```
规划线程主循环:
┌──────────────────────────────────────────────────────────────────┐
│  while not stop_all:                                             │
│                                                                  │
│    1. 等待 need_replan 信号 (阻塞等待，避免空转)                   │
│       need_replan.wait(timeout=0.5)                              │
│       if not need_replan.is_set():                               │
│           continue                                               │
│       need_replan.clear()                                        │
│                                                                  │
│    2. 获取当前机械臂状态                                           │
│       current_joint = env.get_joints()                           │
│                                                                  │
│    3. 获取最新的目标位姿                                           │
│       with state.lock:                                           │
│           target_T = state.latest_target_T.copy()                │
│                                                                  │
│    4. 根据 task_phase 计算完整的目标位姿序列                        │
│       ├── PICK 阶段: [pre_pick_pose, pick_pose, post_pick_pose]  │
│       └── PLACE 阶段: [intermediate, pre_place, place, post_place]│
│                                                                  │
│    5. 调用 curobo 规划                                            │
│       trajectory = curobo.plan(current_joint, target_poses)      │
│                                                                  │
│    6. 转换轨迹为 xyzrpy 动作序列                                   │
│       action_list = trajectory_to_xyzrpy_list(trajectory)        │
│       action_list = downsample(action_list)  # 下采样             │
│                                                                  │
│    7. 更新共享状态（原子操作）                                      │
│       with state.lock:                                           │
│           state.abort_execution.set()   # 先中止旧的执行           │
│           state.action_list = action_list                        │
│           state.action_index = 0                                 │
│           state.execution_done.clear()                           │
│           state.abort_execution.clear() # 允许新的执行开始         │
│           state.plan_ready.set()        # 通知执行线程             │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**关于 curobo 的调用**：
- curobo 的 `plan_single()` 接收当前关节状态和目标 Pose，输出一条关节轨迹
- 需要将关节轨迹通过 FK 转换为 xyzrpy 序列（笛卡尔空间），或直接以关节空间下发
- 建议使用关节空间下发 (`rm_movej_follow`)，更稳定
- 下采样参数建议：每 2~3 个取 1 个，或限制最大点数为 50~80

**重要：规划线程只负责 APPROACH 段**：
- 规划目标始终是 pre_pick 或 pre_place（物体上方安全位置）
- 不需要规划下压/提起（CRITICAL 段），那些由主线程直接下发
- 通过 `state.plan_stage == "APPROACH"` 控制

### 4.3 执行线程（Execution Thread）

**职责**：从动作列表中依次取出动作点，发送给机械臂执行。支持中断和恢复。

```
执行线程主循环:
┌──────────────────────────────────────────────────────────────────┐
│  while not stop_all:                                             │
│                                                                  │
│    1. 等待 plan_ready 信号                                        │
│       plan_ready.wait(timeout=0.5)                               │
│       if not plan_ready.is_set():                                │
│           continue                                               │
│                                                                  │
│    2. 开始执行动作序列                                             │
│       while True:                                                │
│         with state.lock:                                         │
│           # 检查是否被中止                                        │
│           if abort_execution.is_set():                           │
│               plan_ready.clear()                                 │
│               break     # 跳出，等待新的 plan_ready              │
│                                                                  │
│           # 检查是否执行完毕                                      │
│           if action_index >= len(action_list):                   │
│               plan_ready.clear()                                 │
│               execution_done.set()  # 通知主线程                  │
│               break                                              │
│                                                                  │
│           # 取出当前动作点                                        │
│           action = action_list[action_index]                     │
│           action_index += 1                                      │
│                                                                  │
│         # 发送到机械臂（锁外执行，避免长时间持锁）                   │
│         env.arm.rm_movej_follow(action)  # 或 send_pose(action)  │
│                                                                  │
│         sleep(CONTROL_INTERVAL)  # 20ms = 50Hz                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 五、物体移动时的重新规划机制（核心难点）

### 5.1 正常追踪阶段的重规划

在动作序列执行过程中，如果感知线程检测到物体移动了（距离 > 阈值），流程如下：

```
感知线程检测到物体移动
    │
    ├── 1. 设置 abort_execution 信号
    │      → 执行线程收到后立即停止当前动作序列
    │
    ├── 2. 设置 need_replan 信号
    │      → 规划线程收到后根据新的目标位置重新规划
    │
    └── 3. 规划线程完成后：
           ├── 清空旧动作列表
           ├── 写入新动作列表
           ├── 重置 action_index = 0
           └── 设置 plan_ready → 执行线程开始执行新序列
```

### 5.2 pick/place 执行过程中的特殊处理

**关键问题**：pick 的完整动作是「接近 pre_pick → 下压到 pick → 闭合夹爪 → 提起到 post_pick」。在下压和提起的过程中，机械臂自身会遮挡物体、推动物体，导致：
1. VLM 打点结果不可靠（被遮挡/视角变化）
2. 物体坐标发生位移（被夹爪推动、被抓起），但这是**预期行为**
3. 如果感知线程此时仍然打点，会误判为"物体自己移动了"，触发不必要的重规划

**解决方案：将 Pick/Place 拆分为两段子阶段，由主线程分段调度**

核心思想：不是在动作列表内部打 zone 标记让感知线程去判断，而是**从根本上把一次 pick 拆成两段独立的子阶段**，主线程在两段之间显式地控制感知线程的启停。这样更清晰、更可靠、也更容易调试。

```
一次完整的 PICK 流程 = 两段子阶段
═══════════════════════════════════════════════════════════════

【子阶段 1：APPROACH（接近）】   ← 感知+规划+执行 三线程协作
    感知线程: ✅ 活跃，持续打点追踪物体
    规划线程: ✅ 活跃，物体移动时重新规划
    执行线程: ✅ 活跃，执行 curobo 规划的轨迹到 pre_pick 位置
    
    目标: 从当前位置 → 到达物体正上方 (pre_pick)
    允许重规划: ✅ 是（物体在桌上被推走等情况）
    
    结束条件: execution_done（到达 pre_pick）
    
                    ↓ 主线程在此切换 ↓

【子阶段 2：CRITICAL（关键动作）】   ← 仅执行线程工作
    感知线程: ⛔ 暂停（set pause_perception）
    规划线程: ⛔ 无事可做（无 need_replan 信号）
    执行线程: ✅ 执行固定的末端动作序列（不需要 curobo 规划）
    
    动作序列 (短距离直线运动，直接下发位姿):
      1. pre_pick → pick（下压，夹爪张开）
      2. pick 位置闭合夹爪
      3. pick → post_pick（提起，夹爪保持闭合）
    
    允许重规划: ⛔ 否（机械臂遮挡+物体被抓，打点无意义）
    
    结束条件: execution_done（提起完成）

═══════════════════════════════════════════════════════════════
```

**为什么不用 zone 标记而是用两段调度？**

| 方案 | 问题 |
|------|------|
| zone 标记方案 | 感知线程仍在打点，VLM 推理浪费算力；打点结果不可靠（遮挡）会污染 `latest_point_3d`；zone 判断依赖 `action_index` 存在竞态条件 |
| **两段调度方案** ✅ | 感知线程在 CRITICAL 段完全暂停，零开销；不会产生脏数据；主线程的 if-else 分段逻辑一目了然，易于调试 |

**Place 阶段同理**：

```
一次完整的 PLACE 流程 = 两段子阶段
═══════════════════════════════════════════════════════════════

【子阶段 1：APPROACH】   ← 三线程协作
    目标: 从 post_pick → 到达容器上方 (pre_place)
    感知线程追踪容器位置，允许重规划

                    ↓ 主线程切换 ↓

【子阶段 2：CRITICAL】   ← 仅执行
    动作序列:
      1. pre_place → place（下放）
      2. place 位置张开夹爪
      3. place → post_place（抬起）
    感知线程暂停

═══════════════════════════════════════════════════════════════
```

## 六、从持续打点到结果检测的模式切换（核心设计）

### 6.1 模式切换状态机

整个单任务的执行过程可以用以下状态机表示：

```
                    ┌─────────────────────────────────┐
                    │          TRACKING 模式           │
                    │   感知线程: 持续打点              │
                    │   规划线程: 响应 need_replan      │
                    │   执行线程: 执行动作列表          │
                    └──────────────┬──────────────────┘
                                   │
                       execution_done 信号触发
                       (动作列表全部执行完毕)
                                   │
                                   ▼
                    ┌─────────────────────────────────┐
                    │         CHECKING 模式            │
                    │   感知线程: 暂停 (pause)          │
                    │   规划线程: 暂停 (无 replan 信号) │
                    │   主线程:   调用 VLM 检测结果     │
                    └──────────────┬──────────────────┘
                                   │
                         ┌─────────┴─────────┐
                    成功 (Success)       失败 (Failure)
                         │                   │
                    ┌────┴────┐         ┌────┴────┐
                    │ PICK成功 │         │ PICK失败 │
                    │→进入PLACE│         │→重试PICK │
                    │ 阶段     │         │          │
                    │ PLACE成功│         │ PLACE失败│
                    │→RESET   │         │→重试PICK │
                    └─────────┘         └─────────┘
```

### 6.2 主线程的调度逻辑（核心控制器）—— 两段式调度

主线程负责在 APPROACH（三线程协作）和 CRITICAL（仅执行）之间切换，并在 CRITICAL 结束后进行成功/失败检测。

```python
def run_single_task(state, env, rs_env, cam_results, task, home_T_tcp2base):
    """执行单个 pick-and-place 任务（两段式调度）"""

    max_retries = 3
    pick_success = False
    place_success = False

    # ════════════════════════════════════════════
    # PICK 阶段
    # ════════════════════════════════════════════
    for attempt in range(max_retries):
        print(f"🎯 PICK attempt {attempt + 1}: {task['pick']}")

        # ─────────────────────────────────────
        # 子阶段 1: APPROACH（到达 pre_pick）
        #   感知✅ 规划✅ 执行✅
        # ─────────────────────────────────────
        with state.lock:
            state.task_phase = TaskPhase.PICK
            state.system_mode = SystemMode.TRACKING
            state.target_name = task['pick']
            state.is_first_point = True
            state.last_stable_point_3d = None
            state.plan_stage = "APPROACH"       # 告诉规划线程: 只规划到 pre_pick
            state.action_list = []
            state.action_index = 0
            state.execution_done.clear()
            state.abort_execution.clear()
            state.pause_perception.clear()      # 感知线程活跃

        # 等待 APPROACH 段执行完毕（到达 pre_pick）
        state.execution_done.wait()
        print("📍 已到达 pre_pick 位置")

        # ─────────────────────────────────────
        # 子阶段 2: CRITICAL（下压→抓取→提起）
        #   感知⛔ 规划⛔ 仅执行✅
        # ─────────────────────────────────────
        with state.lock:
            state.system_mode = SystemMode.EXECUTING
            state.pause_perception.set()        # ⛔ 暂停感知线程

            # 基于感知线程最后一次的打点结果，计算固定的末端动作序列
            # 这些是短距离直线运动，无需 curobo 规划
            target_T = state.latest_target_T.copy()

        # 构建 CRITICAL 动作序列
        pick_T = target_T                                               # 物体位姿
        pre_pick_T = make_lift_T(pick_T, lift_z=SAFE_HEIGHT)            # 上方
        post_pick_T = make_lift_T(pick_T, lift_z=SAFE_HEIGHT)           # 提起

        critical_actions = [
            {"pose": env.xyzrpy_from_T(pick_T),      "gripper": GRIPPER_OPEN,  "wait": 0.0},
            {"pose": env.xyzrpy_from_T(pick_T),      "gripper": GRIPPER_CLOSE, "wait": 0.5},  # 闭合夹爪，等待
            {"pose": env.xyzrpy_from_T(post_pick_T), "gripper": GRIPPER_CLOSE, "wait": 0.0},
        ]

        # 直接在主线程中阻塞式执行 CRITICAL 序列（短且确定，无需多线程）
        execute_critical_sequence(env, critical_actions)
        print("🤏 CRITICAL 执行完毕（已闭合夹爪并提起）")

        # ─────────────────────────────────────
        # 结果检测: CHECKING
        # ─────────────────────────────────────
        with state.lock:
            state.system_mode = SystemMode.CHECKING
            # 感知线程仍然暂停（pause_perception 仍为 set）

        pick_success = check_pick_success_vllm(rs_env, task['pick'])

        if pick_success:
            print("✅ Pick 成功!")
            break
        else:
            print("❌ Pick 失败, 重试...")
            # 松开夹爪，回到安全位
            env.send_gripper(GRIPPER_OPEN)
            time.sleep(0.3)
            continue

    if not pick_success:
        print("⛔ Pick 多次重试失败，跳过此任务")
        return False

    # ════════════════════════════════════════════
    # PLACE 阶段
    # ════════════════════════════════════════════
    for attempt in range(max_retries):
        print(f"📦 PLACE attempt {attempt + 1}: {task['place']}")

        # ─────────────────────────────────────
        # 子阶段 1: APPROACH（到达 pre_place）
        #   感知✅ 规划✅ 执行✅
        # ─────────────────────────────────────
        with state.lock:
            state.task_phase = TaskPhase.PLACE
            state.system_mode = SystemMode.TRACKING
            state.target_name = task['place']
            state.is_first_point = True
            state.last_stable_point_3d = None
            state.plan_stage = "APPROACH"
            state.action_list = []
            state.action_index = 0
            state.execution_done.clear()
            state.abort_execution.clear()
            state.pause_perception.clear()      # 感知线程恢复

        state.execution_done.wait()
        print("📍 已到达 pre_place 位置")

        # ─────────────────────────────────────
        # 子阶段 2: CRITICAL（下放→松开→抬起）
        #   感知⛔ 规划⛔ 仅执行✅
        # ─────────────────────────────────────
        with state.lock:
            state.system_mode = SystemMode.EXECUTING
            state.pause_perception.set()
            target_T = state.latest_target_T.copy()

        place_T = make_lift_T(target_T, lift_z=0.13)                    # 放置高度
        pre_place_T = make_lift_T(place_T, lift_z=SAFE_HEIGHT)
        post_place_T = make_lift_T(place_T, lift_z=SAFE_HEIGHT)

        critical_actions = [
            {"pose": env.xyzrpy_from_T(place_T),      "gripper": GRIPPER_CLOSE, "wait": 0.0},
            {"pose": env.xyzrpy_from_T(place_T),      "gripper": GRIPPER_OPEN,  "wait": 0.5},  # 松开夹爪
            {"pose": env.xyzrpy_from_T(post_place_T), "gripper": GRIPPER_OPEN,  "wait": 0.0},
        ]

        execute_critical_sequence(env, critical_actions)
        print("📤 CRITICAL 执行完毕（已松开夹爪并抬起）")

        # ─────────────────────────────────────
        # 结果检测
        # ─────────────────────────────────────
        with state.lock:
            state.system_mode = SystemMode.CHECKING

        place_success = check_place_success_vllm(rs_env, task['pick'], task['place'])

        if place_success:
            print("✅ Place 成功!")
            break
        else:
            print("❌ Place 失败, 从 pick 重新开始...")
            env.reset()     # 阻塞式回初始位
            return run_single_task(state, env, rs_env, cam_results, task, home_T_tcp2base)

    if not place_success:
        print("⛔ Place 多次重试失败")
        return False

    return True


def execute_critical_sequence(env, critical_actions):
    """
    阻塞式执行 CRITICAL 动作序列（短距离确定性动作）
    不经过多线程，直接在调用线程中顺序执行
    """
    for action in critical_actions:
        # 运动
        env.send_pose(action["pose"])
        time.sleep(1.0)  # 等待到位（短距离运动）

        # 夹爪
        env.send_gripper(action["gripper"])
        if action.get("wait", 0) > 0:
            time.sleep(action["wait"])  # 夹爪动作等待
```

**两段式调度的关键优势**：

| 特性 | 说明 |
|------|------|
| **感知线程零干扰** | CRITICAL 段感知线程完全暂停，不会产生任何脏数据 |
| **无需 zone 判断** | 主线程显式分段调度，避免了感知线程内部判断 zone 的竞态条件 |
| **CRITICAL 段不需要 curobo** | 下压/提起都是短距离直线运动（几厘米），直接下发目标位姿即可，无碰撞风险 |
| **调试友好** | 每段独立，可以分别测试 APPROACH 和 CRITICAL |
| **安全** | CRITICAL 段在主线程阻塞执行，动作确定，不会被其他线程打断 |

### 7.1 感知线程完整逻辑

```python
PERCEPTION_INTERVAL = 0.3    # 打点频率（秒），取决于 VLM 推理速度
MOVE_THRESHOLD = 0.03        # 物体移动检测阈值（米，3cm）

def perception_thread(state, rs_env, cam_results, home_T_tcp2base):
    """感知线程：持续打点并检测物体位置变化"""
    print("🔍 感知线程已启动")

    while not state.stop_all.is_set():

        # --- 暂停检测 ---
        if state.pause_perception.is_set():
            time.sleep(0.05)
            continue

        # --- 获取当前追踪目标 ---
        with state.lock:
            target_name = state.target_name
            if target_name is None:
                time.sleep(0.1)
                continue

        try:
            # 1. 获取 RGB 图像
            obs = rs_env.step()
            image_rgb = obs["rgb"]

            # 2. 调用 VLM 打点
            result = get_point_vllm(image_rgb, f"Point the {target_name}")
            point_2d = result[0]["point_2d"] if isinstance(result, list) else result

            # 3. 2D → 3D 转换
            target_T = make_target_T(
                obs, int(point_2d[0]), int(point_2d[1]),
                rs_env, cam_results, home_T_tcp2base
            )
            new_xyz = target_T[:3, 3]

            # 4. 更新共享状态并进行变化检测
            with state.lock:
                state.latest_point_2d = point_2d
                state.latest_point_3d = new_xyz.copy()
                state.latest_target_T = target_T.copy()

                if state.is_first_point:
                    # 第一个有效点
                    state.last_stable_point_3d = new_xyz.copy()
                    state.is_first_point = False
                    state.point_changed = True
                    state.need_replan.set()
                    print(f"📍 首次定位 {target_name}: {new_xyz}")

                else:
                    dist = np.linalg.norm(new_xyz - state.last_stable_point_3d)

                    if dist > MOVE_THRESHOLD:
                        # 物体移动了！
                        # 注意：感知线程只在 APPROACH 段活跃（CRITICAL 段已被暂停）
                        # 所以到达这里时一定是在 APPROACH 段，无需额外判断 zone
                        state.last_stable_point_3d = new_xyz.copy()
                        state.point_changed = True
                        state.abort_execution.set()   # 中止当前执行
                        state.need_replan.set()       # 触发重规划
                        print(f"⚠️ 物体 {target_name} 移动! dist={dist:.4f}m → 重规划")
                    else:
                        state.point_changed = False

        except Exception as e:
            print(f"感知异常: {e}")

        time.sleep(PERCEPTION_INTERVAL)

```

### 7.2 规划线程完整逻辑

```python
def planning_thread(state, env, curobo_planner, home_T_tcp2base):
    """规划线程：接收重规划请求，调用 curobo 生成动作序列"""
    print("🧠 规划线程已启动")

    while not state.stop_all.is_set():

        # 等待重规划信号
        triggered = state.need_replan.wait(timeout=0.5)
        if not triggered:
            continue
        state.need_replan.clear()

        # 如果系统不在 TRACKING 模式，跳过
        with state.lock:
            if state.system_mode != SystemMode.TRACKING:
                continue
            target_T = state.latest_target_T
            task_phase = state.task_phase
            if target_T is None:
                continue
            target_T = target_T.copy()

        try:
            # 1. 获取当前关节状态
            current_joint = env.get_joints()

            # 2. 构建 APPROACH 段的动作列表（只规划到 pre_pick/pre_place）
            #    CRITICAL 段（下压/抓取/提起）由主线程单独处理
            action_list = build_approach_action_list(
                target_T, home_T_tcp2base, current_joint,
                curobo_planner, task_phase
            )

            if action_list is None or len(action_list) == 0:
                print("⚠️ 规划失败，等待下次重规划")
                continue

            # 3. 更新共享状态
            with state.lock:
                state.abort_execution.set()      # 中止旧执行
                time.sleep(0.02)                 # 给执行线程一点时间响应
                state.action_list = action_list
                state.action_index = 0
                state.execution_done.clear()
                state.abort_execution.clear()    # 允许新执行
                state.plan_ready.set()           # 通知执行线程
                print(f"📐 规划完成，动作序列长度: {len(action_list)}")

        except Exception as e:
            print(f"规划异常: {e}")


def build_approach_action_list(target_T, home_T_tcp2base, current_joint, 
                               curobo_planner, task_phase):
    """
    构建 APPROACH 段的动作序列（只规划到 pre_pick / pre_place）
    
    CRITICAL 段（下压/抓取/提起）由主线程单独处理，不在此函数中。

    参数:
        target_T: 目标物体的 4x4 位姿矩阵
        task_phase: TaskPhase.PICK 或 TaskPhase.PLACE

    返回格式:
    [
        {"joints": [...], "gripper": GRIPPER_OPEN},
        {"joints": [...], "gripper": GRIPPER_OPEN},
        ...  # curobo 规划的轨迹点，终点为 pre_pick/pre_place
    ]
    """

    # 1. 计算目标：物体上方安全位置
    pre_target_T = make_lift_T(target_T, lift_z=SAFE_HEIGHT)

    # 2. 调用 curobo 规划从当前关节状态到 pre_target 的无碰撞轨迹
    trajectory = curobo_planner.plan(current_joint, pre_target_T)

    if trajectory is None:
        return None

    # 3. 下采样（curobo 输出通常 100~500 点，太密了）
    trajectory = trajectory[::2]

    # 4. 构建动作列表
    gripper_state = GRIPPER_OPEN if task_phase == TaskPhase.PICK else GRIPPER_CLOSE
    action_list = []
    for joints in trajectory:
        action_list.append({
            "joints": joints,
            "gripper": gripper_state
        })

    return action_list
```

### 7.3 执行线程完整逻辑

```python
CONTROL_INTERVAL = 0.02   # 50Hz 控制频率

def execution_thread(state, env):
    """执行线程：依次执行动作列表中的动作点"""
    print("🤖 执行线程已启动")

    while not state.stop_all.is_set():

        # 等待规划完成信号
        triggered = state.plan_ready.wait(timeout=0.5)
        if not triggered:
            continue

        print("▶️ 开始执行动作序列")

        while not state.stop_all.is_set():

            # 检查中止信号
            if state.abort_execution.is_set():
                print("⏹️ 执行被中止，等待新规划")
                state.plan_ready.clear()
                break

            with state.lock:
                # 检查是否执行完毕
                if state.action_index >= len(state.action_list):
                    state.plan_ready.clear()
                    state.execution_done.set()
                    print("✅ 动作序列执行完毕")
                    break

                # 取出当前动作
                action = state.action_list[state.action_index]
                state.action_index += 1

            # === 执行动作（锁外，避免长时间持锁）===

            # 处理夹爪
            if "gripper" in action:
                env.send_gripper(action["gripper"])

            # 处理运动
            if "joints" in action:
                env.arm.rm_movej_follow(action["joints"])
            elif "pose" in action:
                env.send_pose(action["pose"])

            time.sleep(CONTROL_INTERVAL)
```

```python
def main():
```

## 九、线程间同步信号一览

| 信号名 | 类型 | 设置方 | 等待方 | 用途 |
|--------|------|--------|--------|------|
| `need_replan` | Event | 感知线程 | 规划线程 | 目标位置变化，需要重新规划 |
| `plan_ready` | Event | 规划线程 | 执行线程 | 新的动作序列已就绪 |
| `execution_done` | Event | 执行线程 | 主线程 | 动作序列执行完毕 |
| `abort_execution` | Event | 感知线程/规划线程 | 执行线程 | 中止当前执行 |
| `pause_perception` | Event | 主线程 | 感知线程 | 暂停感知（进入检测模式） |
| `stop_all` | Event | 主线程 | 所有线程 | 全局停止 |


**要点说明**：
1. APPROACH 段中物体移动被正确处理（中止+重规划），这段期间感知线程始终活跃
2. 到达 pre_pick 后，主线程设置 `pause_perception`，感知线程**立即暂停**
3. CRITICAL 段由主线程直接阻塞执行（`send_pose` + `send_gripper`），不经过执行线程
4. 整个 CRITICAL 过程中不会有任何 VLM 打点调用，避免了机械臂遮挡导致的误判
5. CRITICAL 完成后进入 CHECKING，感知线程仍然暂停，直到 CHECK 完成后根据结果决定是否恢复

## 十三、注意事项和最佳实践

### 13.1 线程安全
- 所有对 `SharedState` 的读写必须在 `with state.lock` 内完成
- 但机械臂控制命令（`send_pose`, `rm_movej_follow`）必须在锁外执行，否则会造成死锁或长时间阻塞

### 13.2 避免竞态条件
- `abort_execution` 的设置和清除有严格时序：先 set → 等执行线程响应 → 再更新动作列表 → 最后 clear
- 使用 `time.sleep(0.02)` 给执行线程响应时间

### 13.3 curobo 调用注意
- curobo 初始化（warmup）较慢（约 10~30 秒），必须在主线程启动时完成
- curobo 的 `plan_single()` 调用约 30~100ms，不会阻塞太久
- 如果规划失败，不要立即重试，等待下次 `need_replan` 信号
- 规划结果需要下采样，否则执行时间过长

### 13.4 夹爪控制时机
- 夹爪动作（开/合）应在 CRITICAL 区的特定动作点执行
- 夹爪动作本身需要时间（约 0.5 秒），在动作点执行时加入适当延时

### 13.5 异常处理
- 感知线程的 VLM 调用可能失败，需要 try-catch 跳过
- curobo 规划可能失败（IK 无解或碰撞无法避免），保持当前状态不动
- 执行过程中机械臂可能报错，需要捕获并安全停止

## 十四、后续可优化方向

1. **轨迹平滑过渡**：重规划后新旧轨迹的拼接和平滑处理
2. **自适应打点频率**：物体静止时降低频率，移动时提高频率
3. **多相机融合**：使用多个相机提高定位精度
4. **力控反馈**：在 CRITICAL 区加入力传感器反馈，精准检测抓取成功
5. **双臂协作**：扩展为双臂同时执行不同任务
6. **可视化界面**：实时显示各线程状态、物体追踪轨迹和机械臂动作
